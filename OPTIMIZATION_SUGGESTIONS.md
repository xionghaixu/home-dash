# home-dash 后端功能规划与阶段设计文档

> 文档定位：
> 本文件只承担后端功能规划、阶段路线和交付边界说明。
> 开发规范、代码规范、接口约束细则请查看 `SKILLS_SPEC.md`。
> README 保持项目介绍、部署和快速开始，不与本文件混写。

## 1. 项目目标

`home-dash` 后端不是一个单纯“能上传和下载文件”的服务端，而是一个以“个人网盘”为起点，逐步演进到“家庭 NAS”的单体式存储中枢。

### 1.1 当前阶段约束

- 技术主干：JDK 21 + Spring Boot 3.x + MyBatis-Plus
- 架构形态：继续保持单体项目，优先把功能闭环和页面支撑做完整
- 使用场景：单用户、家庭主人、局域网优先
- 产品主线：先解决“存、看、找、整理、治理、备份”五类问题
- 延后主题：登录、权限、安全、域名、SSL、公网暴露统一放到第六阶段

### 1.2 文档边界

- 本文档讨论“做什么、在哪个阶段做、做到什么程度”
- `SKILLS_SPEC.md` 讨论“如何开发、如何约束代码”
- 前端页面与交互设计请查看 `../home-dash-web/OPTIMIZATION_SUGGESTIONS.md`

## 2. 规划原则

### 2.1 阶段独立

每个阶段都必须具备独立排期、开发、联调和验收的条件：

- 阶段内功能形成完整闭环，不把一个能力拆成跨阶段半成品
- 阶段结束后可以形成可演示、可灰度、可继续扩展的版本
- 每个阶段都要同时明确后端接口、数据结构、任务机制和页面支撑

### 2.2 页面驱动

后端设计不能脱离页面和交互：

- 每个阶段都必须明确支撑哪些页面
- 每个接口都应尽量直接服务于页面展示和操作闭环
- 需要异步处理的能力应从一开始就考虑任务化设计

### 2.3 数据前置

项目未来会从个人网盘发展为家庭 NAS，因此数据模型需要为后续能力预留空间：

- 目录树和业务视图持续由 `File` 领域承接
- 物理资源和去重由 `Resource` 领域承接
- 分块上传和断点续传由 `ResourceChunk` 领域承接
- 预览、标签、收藏、媒体信息后续通过扩展元数据承接
- 中长耗时能力统一向任务中心收敛

### 2.4 当前不进入主线的内容

以下主题统一不进入阶段一到阶段五主线：

- 登录 / 注册
- 用户与角色体系
- 权限控制
- 公网访问安全
- 域名接入
- SSL 证书
- 对外分享鉴权
- 限流、防爆破、审计

以上内容统一收敛到第六阶段。

## 4. 核心领域模型规划

| 领域 | 核心对象 | 作用 | 首次引入阶段 |
|------|----------|------|--------------|
| 目录树领域 | `File` | 承载目录树、文件业务视图、移动复制删除等逻辑 | 阶段一 |
| 物理资源领域 | `Resource` | 承载物理文件、去重、下载、媒体源文件 | 阶段一 |
| 分块上传领域 | `ResourceChunk` | 承载分块上传、断点续传、完整性校验 | 阶段一 |
| 元数据领域 | `FileMetadata` | 承载标签、收藏、最近使用、媒体元信息 | 阶段二 |
| 预览资源领域 | `PreviewAsset` | 承载缩略图、封面、文本预览缓存等 | 阶段二 |
| 媒体领域 | `MediaProfile` | 承载时长、分辨率、专辑、艺术家、拍摄时间等 | 阶段三 |
| 任务领域 | `Task` / `TaskExecution` | 承载扫描、缩略图、统计、导入、备份等异步任务 | 阶段四 |
| 治理领域 | `RecycleRecord` / `DuplicateGroup` | 承载回收站和重复文件治理 | 阶段四 |
| 记录领域 | `OperationLog` | 承载关键操作审计和历史回溯 | 阶段四 |
| 设备领域 | `Device` / `ImportSource` | 承载设备接入、目录导入、外部存储来源 | 阶段五 |
| 备份领域 | `BackupPlan` / `SyncRule` | 承载同步、备份计划、执行结果 | 阶段五 |

## 6. 阶段一：核心网盘可用版

### 6.1 阶段目标

阶段一不是“只有上传下载”的演示版，而是要做出一个能实际使用的初步网盘版本。

这个阶段应至少解决以下问题：

- 文件可以稳定地存进去
- 文件可以按目录管理起来
- 文件可以按基础类型和最近上传快速找到
- 用户能看到传输状态和系统基础信息
- 页面只靠少量菜单也能形成一个完整闭环

### 6.2 阶段一建议形成的一级菜单能力

从产品完整度看，阶段一建议后端至少支撑以下入口：

- 全部文件
- 最近上传
- 基础分类浏览
- 传输列表
- 系统信息
- 视频播放页入口

其中“基础分类浏览”在阶段一不要求形成完整媒体中心，但至少应能支持以下基础分类列表：

- 图片
- 视频
- 音频
- 文档
- 压缩包
- 其他文件

## 12. 跨阶段接口与任务设计建议

### 12.1 API 规范

- 保持 `/v1` 前缀统一
- 列表接口统一支持分页、排序、筛选
- 长耗时操作优先任务化
- 统一响应结构，并保留 `extra` 这类扩展字段用于页面补充信息

### 12.2 任务设计建议

- 任务最小字段建议：`id`、`type`、`source`、`status`、`progress`、`resultSummary`、`errorMessage`
- 任务状态建议：`waiting`、`running`、`success`、`failed`、`canceled`
- 所有任务都应可被页面感知，而不是只停留在日志中

### 12.3 存储与元数据建议

- 原文件和预览资源分离存储
- 缩略图、封面、文本缓存统一归入预览资源目录
- 元数据更新优先异步化，避免阻塞上传主链路
- 目录统计、重复检测、媒体扫描尽量后台任务化

## 13. 当前推进建议

按照当前代码现状，后端优先级建议如下：

1. 先把阶段一剩余增强项补齐，尤其是秒传正式接口、上传状态语义和冲突命名策略。
2. 之后进入阶段二，优先建设搜索、详情、图片 / 文本 / 音频预览。
3. 阶段三再把首页、图片、视频、音频三类媒体能力整体拉起来。
4. 阶段四和阶段五分别承接治理能力与家庭设备联动能力。

这样可以保证路线清晰，也能最大化复用当前已经完成的一阶段成果。
